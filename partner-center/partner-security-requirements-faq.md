# <a name="frequently-asked-questions-about-the-partner-security-requirements"></a>Часто задаваемые вопросы о требованиях безопасности партнеров

Эта статья содержит некоторые часто задаваемые вопросы о [требованиях безопасности партнеров](partner-security-requirements.md).

## <a name="partner-security-requirements"></a>Требования к безопасности партнеров

### <a name="what-are-the-new-partner-security-requirements"></a>Каковы новые требования к безопасности партнеров?

Чтобы защитить наших партнеров и клиентов, необходимо немедленно предпринять следующие действия для партнеров:  

1. **Включите многофакторную проверку подлинности (MFA) для всех пользователей в партнерских клиентах**. Все пользователи в партнерских клиентах должны использовать многофакторную проверку подлинности (MFA) при входе в коммерческие облачные службы Майкрософт или в CSP с помощью центра партнеров или интерфейсов API. Благодаря включению политик базового уровня защиты MFA вы бесплатно доступны для всех пользователей партнерских клиентов.

2. **Внедрение платформы безопасной модели приложений**. Все партнеры, которые интегрируются с API Майкрософт, например Azure Resource Manager, Microsoft Graph и API центра партнеров, должны принять платформу безопасной модели приложений, чтобы избежать сбоев при их интеграции, если включены базовые политики. 
 
Включение многофакторной идентификации (MFA) и внедрение инфраструктуры защищенных моделей приложений помогут защитить инфраструктуру и защитить данные клиента от потенциальных угроз безопасности, таких как идентификация кражи или другие инциденты мошенничества.  

### <a name="which-partners-need-to-meet-the-requirements"></a>Какие партнеры должны удовлетворять этим требованиям?

Эти требования относятся к следующим группам партнеров:
- Все партнерские организации, участвующие в программе поставщика облачных решений (CSP), взаимодействующие с помощью коммерческих облачных служб Майкрософт
  - Партнеры прямых счетов
  - Косвенные поставщики
  - Непрямые торговые посредники
- Все поставщики панели управления
- Все партнеры по программам Advisor  

Все партнеры, взаимодействующие через облако независимых (21Vianet, правительство США и Германия), не обязаны отвечать новым требованиям безопасности, начиная с 1 августа. Однако настоятельно рекомендуется, чтобы все партнеры, использующие независимых Cloud, и приступали к этим новым требованиям безопасности немедленно. Корпорация Майкрософт предоставит дополнительные сведения о применении этих требований безопасности для облаков независимых в будущем.

### <a name="what-are-the-key-timelines-and-milestones"></a>Каковы ключевые временные шкалы и вехи?

Термины, связанные с этими требованиями к безопасности, будут добавлены непосредственно в руководством по программе для поставщика облачных решений. Вам потребуется реализовать эти требования к безопасности, чтобы обеспечить соответствие вашим участию в программе CSP, действующей с 1 августа 2019. 

### <a name="what-will-happen-if-i-do-not-take-any-actions"></a>Что произойдет, если я не делаю никаких действий?

Партнеры, которые не придерживаться этих методов и обязательств по обеспечению безопасности, не смогут выполнить инструкцию по работе с поставщиком облачных решений или управлять клиентами клиентов с помощью делегированных прав администратора, как только эти требования к безопасности партнеров будут применены. Мы намерены установить дату принудительного применения требований и уведомить партнеров по датам с подробными сведениями.

### <a name="why-is-microsoft-enforcing-these-new-requirements"></a>Почему корпорация Майкрософт принудительно применяет эти новые требования?
Безопасность и конфиденциальность клиентов и партнеров — высший приоритет корпорации Майкрософт. Мы продолжаем видеть более сложные и увеличивающие число атак безопасности, в основном связанных с нарушениями подлинности. По мере того как профилактическые элементы управления играют ключевую роль в общей стратегии защиты, чтобы предотвратить атаки, связанные с безопасностью, мы начнем применить набор обязательных требований безопасности для защиты партнеров и их клиентов.

### <a name="does-this-apply-to-all-geographies"></a>Применяется ли это ко всем географическим регионам?

Да, это относится ко всем географическим регионам. Настоятельно рекомендуется, чтобы все партнеры, взаимодействующие через облако независимых (21Vianet, правительство США и Германия), работали с этими новыми требованиями к безопасности немедленно. Однако эти партнеры не обязаны отвечать новым требованиям безопасности, действующим 1 августа. Корпорация Майкрософт предоставит дополнительные сведения о применении этих требований безопасности для облаков независимых в будущем.

## <a name="required-actions"></a>Необходимые действия

### <a name="what-are-the-key-actions-i-need-to-take-to-meet-the-requirements"></a>Какие ключевые действия необходимо предпринять для удовлетворения требований?  
Все партнеры в программе CSP (прямые счета, косвенные поставщики и непрямые торговые посредники), помощники и поставщики панели управления должны соответствовать требованиям.

1. **Применение MFA для всех пользователей**

    Все партнеры в программе CSP, Advisor и поставщики панели управления должны применять MFA для всех пользователей в клиенте-партнере. Это можно сделать, включив параметр [требовать MFA для администраторов](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-administrators), [базовую защиту конечных пользователей](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-end-users)и любые будущие базовые политики. Функциональные возможности, предоставляемые базовыми политиками, продолжат развиваться, чтобы обеспечить защиту партнеров и клиентов от постоянно меняющихся угроз безопасности. Поэтому для получения дополнительных сведений важно ознакомиться с документацией по [базовым политикам](https://docs.microsoft.com/azure/active-directory/conditional-access/concept-baseline-protection) .

    - См [. раздел Базовая политика: Обязательное использование MFA для](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-administrators) администраторов для получения дополнительных сведений о включении политики "требовать MFA для администратора".
    - См [. раздел Базовая политика: Защита](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-end-users) конечных пользователей для получения дополнительных сведений о том, как включить базовую политику защиты конечных пользователей.
    - Изучите концепцию [политик базовой защиты](https://docs.microsoft.com/azure/active-directory/conditional-access/concept-baseline-protection).

    Дополнительные рекомендации.

    - Косвенные поставщики должны работать с непрямыми торговыми посредниками для перехода в центр партнеров, если они еще не сделали этого, и не рекомендовали своих торговых посредников в соответствии с требованиями.
    - Azure MFA предоставляется всем пользователям в клиенте-партнере бесплатно с помощью базовых политик с единственным методом проверки использования [приложения Microsoft Authenticator](https://docs.microsoft.com/azure/active-directory/user-help/user-help-auth-app-overview).
    - Дополнительные методы проверки доступны в [Azure Active Directory Premium](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-get-started-premium) SKU, если требуются другие методы, такие как SMS или электронная почта.
    - Партнеры могут также использовать стороннее решение MFA для каждого пользователя при доступе к коммерческим облачным службам Майкрософт.

2. **Внедрение платформы безопасной модели приложений**

    Все партнеры, которые разработали пользовательскую интеграцию с помощью любых API (таких как Azure Resource Manager, Microsoft Graph, API центра партнеров и т. д.) или реализуют пользовательскую автоматизацию с помощью таких средств, как PowerShell, должны использовать [модель защищенных приложений. Платформа](https://docs.microsoft.com/partner-center/develop/enable-secure-app-model) для интеграции с облачными службами Майкрософт. Если этого не сделать, это может привести к сбою из-за развертывания MFA. Следующие ресурсы содержат общие сведения и рекомендации по внедрению модели.

    - [Общие сведения о безопасной модели приложений](https://docs.microsoft.com/partner-center/develop/enable-secure-app-model)
    - [Центр партнеров: Рекомендации по безопасной модели приложений](http://assetsprod.microsoft.com/secure-application-model-guide.pdf)
    - [Партнеры в программе CSP: пример кода .NET для включения безопасной модели приложений](http://github.com/microsoft/Partner-Center-DotNet-Samples/tree/master/secure-app-model)
    - [Партнеры в программе CSP: Пример кода Java для включения безопасной модели приложений](http://github.com/microsoft/Partner-Center-Java-Samples/tree/master/secure-app-model)
    - [Документ по проверке подлинности центра партнеров](https://docs.microsoft.com/partner-center/develop/partner-center-authentication)
    - [Документ по многофакторной проверке подлинности PowerShell (MFA) в центре партнеров](https://docs.microsoft.com/partner-center/develop/multi-factor-auth)

    При использовании панели управления необходимо обратиться к поставщику по поводу внедрения инфраструктуры защищенной модели приложений.

    Поставщики панели управления необходимы для [подключения](https://docs.microsoft.com/partner-center/enroll-as-cpv) к центру партнеров в качестве поставщика панели управления и немедленно приступить к реализации этого требования. Обратитесь к [центру партнеров: Безопасная платформа](http://assetsprod.microsoft.com/secure-application-model-guide.pdf)модели приложений. Поставщики панели управления должны принять согласие партнеров CSP и управлять им вместо учетных данных и очищать все существующие учетные данные партнеров CSP.

## <a name="multi-factor-authentication"></a>Многофакторная идентификация

### <a name="what-is-mfa"></a>Что такое MFA?

Многофакторная идентификация (MFA) — это механизм безопасности, в котором пользователи проходят проверку подлинности с помощью нескольких обязательных процедур безопасности и проверки. Для этого требуются два или более следующих метода проверки подлинности:

- Известные вам сведения (обычно это пароль)
- Имеющееся у вас оборудование (доверенное устройство, которое трудно дублировать, например телефон)
- Ваши биологические признаки (биометрические данные)

### <a name="what-are-baseline-protection-policies"></a>Что такое политики защиты базовых показателей? 

[Политики Microsoft Baseline Protection](https://docs.microsoft.com/azure/active-directory/conditional-access/concept-baseline-protection) (сейчас Предварительная версия) — это набор стандартных политик, которые помогают защитить организации от многих распространенных атак. Эти распространенные атаки могут включать распыление паролей, воспроизведение и фишинг. Базовые политики доступны во всех выпусках Azure Active Directory. Корпорация Майкрософт делает эти политики базовой защиты доступными всем, поскольку атаки на основе удостоверений находились в течение последних нескольких лет. Цель этих политик — убедиться, что для всех организаций включен базовый уровень безопасности без дополнительных затрат.

> [!NOTE]
> Базовые политики (Майкрософт) и связанные функции будут продолжать развиваться, чтобы улучшить защиту партнеров и клиентов от постоянно меняющихся угроз безопасности. В ближайшее время могут возникнуть некоторые изменения именования и таксономии с базовыми политиками. Настоятельно рекомендуется непосредственно посетить страницы базовых политик, чтобы ознакомиться с последними сведениями.

### <a name="what-baseline-policies-must-i-enable"></a>Какие базовые политики необходимо включить?

Если вы планируете использовать текущие политики базовой защиты для предоставления MFA каждому пользователю в партнерской клиентской организации, необходимо включить [MFA для администраторов](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-administrators) и базовых политик [защиты конечных пользователей](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-end-users) . Эти политики базовой защиты выполнит требование для MFA для каждого пользователя в клиенте партнера бесплатно только для партнеров, использующих Microsoft Authenticator приложения через мобильное устройство.

[Базовая политика "требовать MFA для администраторов"](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-administrators) используется для администраторов в каталоге партнера, а базовая политика [защиты конечных пользователей](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-end-users) — для защиты пользователей, не имеющих прав администратора, в клиенте партнера. Для включения этих политик пользователям потребуется зарегистрироваться для использования MFA. После успешной регистрации пользователь будет получать запрос на MFA во время попытки входа в зависимости от критериев политики. Функциональные возможности, предоставляемые базовыми политиками, продолжат развиваться, чтобы обеспечить защиту партнеров и клиентов от постоянно меняющихся угроз безопасности. Поэтому для получения дополнительных сведений важно ознакомиться с документацией по [базовым политикам](https://docs.microsoft.com/azure/active-directory/conditional-access/concept-baseline-protection) .

### <a name="how-do-i-enable-the-require-mfa-for-admins-policy"></a>Разделы справки включить политику "требовать MFA для администраторов"? 

Базовую политику "требовать MFA для администраторов" можно включить на портале управления Azure. См [. раздел Базовая политика: ](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-administrators) Для получения дополнительных сведений о том, как включить эту базовую политику, требуется указать MFA для администраторов.

### <a name="how-do-i-enable-the-end-user-protection-policy"></a>Разделы справки включить политику защиты конечных пользователей?

Базовую политику защиты конечных пользователей можно включить на портале управления Azure. См [. раздел Базовая политика: Защита](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-end-users) конечных пользователей для получения дополнительных сведений о том, как включить эту базовую политику.

### <a name="will-the-baseline-policies-be-automatically-enabled"></a>Будут ли включены базовые политики автоматически? 

Нет. для включения этих политик пользователь, являющийся членом роли "Глобальный администратор", "администратор безопасности" или "Администратор условного доступа", должен будет настроить политики для немедленного использования политики.

### <a name="what-is-the-cost-of-enabling-mfa"></a>Какова стоимость включения MFA?

Корпорация Майкрософт предоставляет MFA без каких-либо затрат за счет реализации политик защиты учетных записей [MFA для администраторов](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-administrators) и [конечных пользователей](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-end-users) . Единственным вариантом проверки, доступным в этой версии MFA, является [приложение Microsoft Authenticator](https://docs.microsoft.com/azure/active-directory/user-help/user-help-auth-app-overview). Если требуется телефонный звонок или SMS-сообщение, необходимо приобрести лицензию на [Azure Active Directory Premium](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-get-started-premium) . Кроме того, вы можете использовать стороннее решение для предоставления MFA каждому пользователю в партнерской клиентской организации. в этом случае вы обязаны убедиться, что решение MFA установлено и соответствует требованиям.

### <a name="if-i-already-have-an-mfa-solution-what-actions-do-i-need-to-take"></a>Какие действия необходимо предпринять, если у меня уже есть решение MFA?

С помощью этих требований безопасности пользователи в клиенте-партнере должны пройти проверку подлинности с помощью MFA при доступе к коммерческим облачным службам Майкрософт. Для выполнения этих требований можно использовать стороннее решение. Корпорация Майкрософт больше не предоставляет проверочное тестирование для независимых поставщиков удостоверений, чтобы обеспечить совместимость с Azure Active Directory. Если вы хотите протестировать продукт для взаимодействия, ознакомьтесь с этими [рекомендациями](https://www.microsoft.com/download/details.aspx?id=56843).

### <a name="what-verification-method-can-i-use-to-authenticate-mfa"></a>Какой метод проверки можно использовать для проверки подлинности MFA?

Корпорация Майкрософт предоставляет MFA без каких-либо затрат за счет реализации политик защиты учетных записей [MFA для администраторов](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-administrators) и [конечных пользователей](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-end-users) . Единственным вариантом проверки, доступным в этой версии MFA, является [приложение Microsoft Authenticator](https://docs.microsoft.com/azure/active-directory/user-help/user-help-auth-app-overview). Если требуется телефонный звонок или SMS-сообщение, необходимо приобрести лицензию на [Azure Active Directory Premium](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-get-started-premium) . Кроме того, вы можете использовать стороннее решение для предоставления MFA каждому пользователю в партнерской клиентской организации. в этом случае вы обязаны убедиться, что решение MFA установлено и соответствует требованиям.

### <a name="i-use-multiple-partner-tenants-to-transact-do-i-need-to-implement-mfa-on-them-all"></a>Я использую несколько клиентов-партнеров для Transact. Нужно ли реализовать MFA для всех?

Да, необходимо применить MFA для каждого клиента Azure Active Directory, связанного с программой CSP или программой Advisor. Если вы планируете приобрести лицензию на Azure Active Directory Premium, необходимо приобрести лицензию для пользователя в каждом клиенте Azure Active Directory.

### <a name="does-each-user-in-my-partner-tenant-need-to-have-mfa-enforced"></a>Должен ли каждый пользователь в клиенте-партнере принудительно применять MFA? 

Политика [обязательного использования MFA для администраторов](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-administrators) и [защиты конечных пользователей](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-end-users) с базовой защитой будет применять MFA для каждого пользователя в клиенте партнера. Если вы используете эти политики для предоставления MFA и используете [Microsoft Authenticator](https://docs.microsoft.com/azure/active-directory/user-help/user-help-auth-app-overview) приложение, нет необходимости покупать дополнительные лицензии. В противном случае необходимо приобрести соответствующее решение для предоставления MFA каждому пользователю в клиенте партнера.

### <a name="i-am-a-direct-bill-partner-with-microsoft-what-do-i-need-to-do"></a>Я являюсь прямым партнером по оплате с корпорацией Майкрософт. Что мне нужно сделать?

Партнеры поставщика облачных решений с прямым выставлением счетов должны применять MFA для каждого пользователя в своем клиенте-партнере.

### <a name="i-am-an-indirect-reseller-and-only-transact-though-a-distributor-do-i-still-have-to-do-this"></a>Я не являюсь прямым торговым посредником, а только с распространителем. Нужно ли мне все это сделать?

Все непрямые торговые посредники должны применять MFA для каждого пользователя в своем клиенте-партнере. Это действие, которое должен выполнить непрямой Торговый посредник.

### <a name="i-do-not-use-the-partner-center-api-do-i-still-need-to-implement-mfa"></a>Я не использую API центра партнеров. Нужно ли по-прежнему реализовывать MFA?

Да, это требование безопасности предназначено для всех пользователей, включая администраторов партнеров и конечных пользователей в клиенте-партнере.

### <a name="which-third-party-vendors-provide-mfa-solutions-compatible-with-azure-active-directory"></a>Какие сторонние поставщики предоставляют решения MFA, совместимые с Azure Active Directory?

Существует множество независимых проверок решений MFA в сети, например [Gartner](https://www.gartner.com/en/webinars/3881781). При просмотре поставщиков и решений MFA, партнеры должны убедиться, что выбранное решение совместимо с Azure Active Directory.

Корпорация Майкрософт больше не предоставляет проверочное тестирование для независимых поставщиков удостоверений, чтобы обеспечить совместимость с Azure Active Directory. Если вы хотите протестировать продукт для взаимодействия, ознакомьтесь с этими [рекомендациями](https://www.microsoft.com/download/details.aspx?id=56843).

Дополнительные сведения см. в [списке совместимости Федерации Azure AD](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-fed-compatibility).

### <a name="how-can-i-test-mfa-in-our-integration-sandbox"></a>Как можно протестировать MFA в песочнице интеграции?

Для клиента "песочницы" интеграции должны быть включены политики [обязательного использования MFA для администраторов](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-administrators) и [защиты конечных пользователей](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-end-users) . С помощью этой политики каждый пользователь в клиенте должен пройти проверку подлинности с помощью MFA.

### <a name="will-enabling-mfa-effect-how-i-interact-with-my-customers-tenant"></a>Будет ли включено использование MFA, как я могу взаимодействовать с клиентом клиента? 

Нет. Выполнение этих требований безопасности не повлияет на то, как вы будете управлять клиентами. Возможность выполнения делегированных административных операций не будет прервана.

### <a name="are-my-customers-subject-to-the-partner-security-requirements"></a>Подвержены ли мои клиенты требованиям безопасности партнеров?

Нет, вы не обязаны применять MFA для каждого пользователя в клиентах Azure AD клиента. Однако рекомендуется работать с каждым клиентом, чтобы определить оптимальную защиту пользователей.

### <a name="can-app-passwords-be-used-with-the-baseline-protection-policies"></a>Можно ли использовать пароли приложений с политиками базовой защиты?

Да, можно использовать [пароли приложений](https://docs.microsoft.com/azure/active-directory/authentication/howto-mfa-mfasettings#app-passwords) . Ознакомьтесь с замечаниями по использованию паролей приложений, описанными [здесь](https://docs.microsoft.com/azure/active-directory/authentication/howto-mfa-mfasettings#considerations-about-app-passwords) , чтобы определить, поддерживаются ли они в вашей потребности.

### <a name="can-any-user-be-excluded-from-this-requirement"></a>Можно ли исключить любого пользователя из этого требования? 

Нет, каждый пользователь, включая учетные записи служб, в партнерской клиенте должен пройти проверку подлинности с помощью MFA.

### <a name="do-the-partner-security-requirements-apply-to-the-integration-sandbox"></a>Применяются ли требования к безопасности партнеров к песочнице интеграции?

Да, требования к безопасности партнеров применяются к песочнице интеграции. Это означает, что необходимо реализовать соответствующее решение MFA для пользователей в клиенте песочницы интеграции. Рекомендуется реализовать политики базовой защиты для предоставления MFA.

### <a name="how-do-i-configure-an-emergency-access-break-glass-account"></a>Разделы справки настроить учетную запись аварийного доступа (break)?

Рекомендуется создать одну или две учетные записи аварийного доступа, чтобы предотвратить непреднамеренное блокирование клиента Azure AD. В соответствии с требованиями безопасности партнеров необходимо, чтобы каждый пользователь прошел проверку подлинности с помощью MFA. Это означает, что вам потребуется изменить определение учетной записи для аварийного доступа. Это может быть учетная запись, которая использует стороннее решение для MFA.

### <a name="how-will-guest-users-be-impacted-by-the-partner-security-requirements"></a>Каким будет воздействие гостевых пользователей на требования к безопасности партнеров?

Гостевым пользователям потребуется пройти проверку подлинности с помощью MFA при доступе к ресурсам в клиенте партнера. Требования к безопасности партнеров не влияют на то, что гостевой пользователь будет обращаться к ресурсам в своем клиенте.

### <a name="if-i-am-using-a-third-party-solution-is-active-directory-federation-service-adfs-required"></a>Если я использую стороннее решение Active Directory служба федерации (ADFS)? 

Нет, Active Directory служба федерации (ADFS) не требуется, если используется стороннее решение. Рекомендуется работать с поставщиком решения, чтобы определить требования к решению.

### <a name="is-it-a-requirement-to-enable-the-baseline-protection-policies"></a>Требуется ли включить политики базовой защиты?

Нет, не требуется включать политики базовой защиты. Единственным требованием является принудительное применение MFA для каждого пользователя, включая учетные записи служб, в клиенте партнера.

### <a name="can-conditional-access-be-used-to-meet-the-mfa-requirement"></a>Можно ли использовать условный доступ для удовлетворения требования MFA?

Да, вы можете использовать условный доступ для применения MFA для каждого пользователя, включая учетные записи служб, в клиенте партнера. Тем не менее, учитывая природу с высоким уровнем привилегий партнера, необходимо убедиться, что каждый пользователь имеет запрос MFA для каждой отдельной проверки подлинности. Это означает, что вы не сможете использовать функцию условного доступа, которая обходит требование для MFA.

### <a name="what-verification-options-are-provided-through-the-implementation-of-the-baseline-protection-policies"></a>Какие варианты проверки предоставляются через реализацию политик базовой защиты? 

В отношении версии MFA, доступной при реализации политик базовой защиты, доступен единственный вариант проверки — приложение для проверки подлинности. Использование телефонного звонка и сообщения с текстовым сообщением считается менее безопасным. Таким образом, эти параметры недоступны в этой версии MFA.

### <a name="will-the-service-account-used-by-azure-ad-connect-be-impacted-by-the-partner-security-requirements"></a>Будут ли учетные записи службы, используемые Azure AD Connect, затронуты требованиями безопасности партнеров?

Нет, требования безопасности партнеров не повлияют на учетную запись службы, используемую Azure AD Connect. Если возникла неполадка с Azure AD Connect в результате применения MFA, отправьте запрос на техническую поддержку с помощью службы поддержки Майкрософт.

## <a name="secure-application-model"></a>Безопасная модель приложений

Корпорация Майкрософт представляет собой безопасную и масштабируемую платформу для проверки подлинности партнеров поставщика облачных решений (CSP) и поставщиков панели управления (КПВ), использующих многофакторную проверку подлинности. Дополнительные сведения см. в [инструкции по безопасной модели приложений](http://assetsprod.microsoft.com/secure-application-model-guide.pdf) . Все партнеры, которые разработали пользовательскую интеграцию с помощью любых API (таких как Azure Resource Manager, Microsoft Graph, API центра партнеров и т. д.) или реализуют пользовательскую автоматизацию с помощью таких средств, как PowerShell, должны использовать [модель защищенных приложений. Платформа](https://docs.microsoft.com/partner-center/develop/enable-secure-app-model) для интеграции с облачными службами Майкрософт.

### <a name="what-is-the-secure-application-model"></a>Что такое модель безопасного приложения?

Корпорация Майкрософт представляет собой безопасную и масштабируемую платформу для проверки подлинности партнеров поставщика облачных решений (CSP) и поставщиков панели управления (КПВ), использующих многофакторную проверку подлинности. Дополнительные сведения см. в [инструкции по безопасной модели приложений](http://assetsprod.microsoft.com/secure-application-model-guide.pdf) .

Все партнеры, которые разработали пользовательскую интеграцию с помощью любых API (таких как Azure Resource Manager, Microsoft Graph, API центра партнеров и т. д.) или реализуют пользовательскую автоматизацию с помощью таких средств, как PowerShell, должны использовать [модель защищенных приложений. Платформа](https://docs.microsoft.com/partner-center/develop/enable-secure-app-model) для интеграции с облачными службами Майкрософт. Если этого не сделать, это может привести к сбою из-за развертывания MFA. Следующие ресурсы содержат общие сведения и рекомендации по внедрению модели.

### <a name="who-is-a-control-panel-vendor-cpv"></a>Кто является поставщиком панели управления (КПВ)? 
Поставщик панели управления — это независимый поставщик программного обеспечения, который разрабатывает приложения для использования партнерами CSP для интеграции с API-интерфейсами центра партнеров. Поставщик панели управления не является партнером CSP с прямым доступом к панели мониторинга или интерфейсам API центра партнеров. Подробное описание доступно в [центре партнеров: Пошаговое руководством](http://assetsprod.microsoft.com/secure-application-model-guide.pdf)по модели защищенных приложений.

### <a name="does-the-secure-application-model-need-to-be-implemented-for-the-partner-center-apisdk-only"></a>Необходимо ли реализовать защищенную модель приложений только для API центра партнеров или пакета SDK? 

После того как *требуется включить MFA для администраторов* и базовые политики *защиты конечных пользователей* , каждый пользователь должен пройти проверку подлинности с помощью многофакторной проверки подлинности. Это означает, что необходимо реализовать модель безопасного приложения для каждого API, интерфейса командной строки и модуля PowerShell (например, Azure, Azure AD, MS Online, центра партнеров и т. д.), предназначенного для запуска в неинтерактивном режиме, и зависит от использования учетных данных пользователя для проверки подлинности.

### <a name="i-am-using-automation-tools-such-as-powershell-how-do-i-implement-the-secure-application-model"></a>Я использую средства автоматизации, такие как PowerShell. Разделы справки реализовать модель безопасного приложения?  

Если служба автоматизации предназначена для работы в неинтерактивном режиме и использует учетные данные пользователя для проверки подлинности, необходимо реализовать модель безопасного приложения. См. раздел [Безопасная модель приложений | ](https://docs.microsoft.com/powershell/partnercenter/secure-app-model?view=partnercenterps-1.5)Руководство по реализации этой платформы в PowerShell центра партнеров.  Обратите внимание, что не все средства автоматизации предоставляют возможность проверки подлинности с помощью маркеров доступа. Если вам требуется помощь в понимании того, какие изменения необходимо внести, опубликуйте сообщение в группе [руководств по безопасности центра партнеров](https://www.microsoftpartnercommunity.com/t5/Partner-Center-Security-Guidance/ct-p/partner-center-security-guidance) .

### <a name="what-user-credentials-should-the-application-administrator-provide-when-performing-the-consent-process"></a>Какие учетные данные пользователя должен предоставить администратор приложения при выполнении процесса согласия? 

Рекомендуется использовать учетную запись службы, которой были назначены разрешения с минимальными привилегиями. По отношению к API центра партнеров это означает, что следует использовать учетную запись, которой назначена роль агента продаж или агентов администрирования.

### <a name="why-should-the-application-administrator-not-provide-global-admin-user-credentials-when-performing-the-consent-process"></a>Почему администратор приложения не должен предоставлять учетные данные пользователя глобального администратора при выполнении процесса согласия?

Рекомендуется использовать наименьший уровень привилегий, чтобы определить, каким образом вы уменьшаете риск. Не рекомендуется использовать учетную запись с правами глобального администратора, так как это предоставляет больше разрешений, чем требуется.

### <a name="i-am-a-csp-partner-how-do-i-know-if-my-control-panel-vendor-cpv-is-working-on-implementing-the-solution-or-not"></a>Я являюсь партнером CSP. Разделы справки понять, работает ли поставщик панели управления (КПВ) над реализацией решения? 

Для партнеров, использующих решение "поставщик панели управления (КПВ)" для языка Transact в программе поставщика облачных решений (CSP), вы обязаны обратиться за помощью к КПВ. 

### <a name="i-am-a-cpv-how-do-i-enroll"></a>Я КПВ. Разделы справки зарегистрироваться? 

Чтобы зарегистрироваться в качестве поставщика панели управления (КПВ), следуйте приведенным [здесь](https://docs.microsoft.com/partner-center/enroll-as-cpv)рекомендациям.

Чтобы получить ссылку на регистрацию, КПВС должен связаться [CPVHelp@microsoft.com](mailto:CPVHelp@microsoft.com) с и предоставить спонсора сотрудника Майкрософт, который имеет деловые отношения с КПВ или знает свой бизнес. Например, менеджер по разработке партнеров (PDM).

После регистрации в центре партнеров и регистрации приложений вы получите доступ к API-интерфейсам центра партнеров. Если вы являетесь новым КПВ, вы получите сведения о песочнице через уведомление центра партнеров. После завершения регистрации в качестве Microsoft КПВ и принятия соглашения КПВ вы можете:

1. Управление приложением с несколькими клиентами (Добавление приложений в портал Azure, регистрация и Отмена регистрации приложений в центре партнеров). Примечание. КПВС должны зарегистрировать свои приложения в центре партнеров, чтобы получить разрешение для API-интерфейсов центра партнеров. Добавление приложений только в портал Azure не авторизует приложения КПВ для API-интерфейсов центра партнеров.
2. Просматривайте профиль КПВ и управляйте им.
3. Просматривайте пользователей, которым требуется доступ к возможностям КПВ, и управляйте ими. Единственной ролью, которую может иметь КПВ, является Глобальный администратор.

### <a name="i-am-using-the-partner-center-sdk-will-sdk-automatically-adopt-the-secure-application-model"></a>Я использую пакет SDK для центра партнеров. Будет ли пакет SDK автоматически внедрять модель защищенного приложения? 

Нет, необходимо следовать рекомендациям, приведенным в [руководстве по безопасной модели приложений](http://assetsprod.microsoft.com/secure-application-model-guide.pdf).

### <a name="can-i-generate-a-refresh-token-for-the-secure-application-model-with-accounts-that-do-not-have-mfa-enabled"></a>Можно ли создать маркер обновления для защищенной модели приложений с учетными записями, для которых не включена поддержка MFA?

Да, маркер обновления может быть создан с использованием учетной записи, не имеющей принудительного использования MFA. Однако это не следует делать, поскольку любой токен, созданный с помощью учетной записи, для которой не включена поддержка MFA, не сможет получить доступ к ресурсам из-за необходимости в MFA.

### <a name="how-should-my-application-obtain-an-access-token-if-we-enable-mfa"></a>Как приложение должно получать маркер доступа при включении MFA?

Необходимо следовать [руководству по безопасной модели приложений](http://assetsprod.microsoft.com/secure-application-model-guide.pdf) , в котором подробно описывается, как это сделать, чтобы обеспечить соответствие новым требованиям безопасности. Пример кода .NET можно найти [здесь](http://github.com/microsoft/Partner-Center-DotNet-Samples/tree/master/secure-app-model) и в примере кода Java [здесь](http://github.com/microsoft/Partner-Center-Java-Samples/tree/master/secure-app-model).

### <a name="as-a-cpv-do-i-create-an-azure-ad-application-in-our-cpv-tenant-or-the-tenant-of-the-csp-partner"></a>Как КПВ, я создаю приложение Azure AD в нашем клиенте КПВ или в клиенте партнера CSP?

КПВ потребуется создать в клиенте приложение Azure Active Directory, связанное с регистрацией как КПВ.

### <a name="i-am-a-csp-that-is-using-app-only-authentication-do-i-need-to-make-any-changes"></a>Я использую CSP, использующий только проверку подлинности приложения. Нужно ли вносить изменения?

Проверка подлинности только для приложения не влияет на то, что учетные данные пользователя не используются для запроса маркера доступа. Если учетные данные пользователя являются общими, то поставщики панели управления (КПВС) должны принять [платформу безопасной модели приложений](http://assetsprod.microsoft.com/secure-application-model-guide.pdf) и удалить все существующие учетные данные партнера.

### <a name="as-a-cpv-can-i-leverage-the-app-only-authentication-style-to-get-access-tokens"></a>Как КПВ можно использовать стиль проверки подлинности приложения только для получения маркеров доступа?

Нет, партнеры поставщиков панели управления не могут использовать стиль проверки подлинности "только приложение" для запроса маркеров доступа от имени партнера. Они должны реализовывать безопасную модель приложений, использующую стиль проверки подлинности "приложение + пользователь".

## <a name="support"></a>Поддержка

### <a name="where-can-i-ask-get-support"></a>Где можно задать поддержку для получения поддержки?

Кроме того, для поддержки использования ресурсов в соответствии с требованиями безопасности, если у вас есть Расширенная поддержка для партнеров (ASfP), обратитесь к диспетчеру учетных записей служб. для поддержка Premier по соглашению для партнеров (PSfP) обратитесь к менеджеру по обслуживанию и менеджеру по технической учетной записи.

### <a name="how-can-i-get-help-with-enabling-the-baseline-policies"></a>Как получить справку по включению базовых политик?

- Партнеры могут использовать часы из преимуществ MPN, чтобы получить более подробные рекомендации по реализации требований безопасности.
- Технические варианты поддержки для Azure Active Directory доступны по преимуществам MPN. Партнеры с доступом к активным соглашениям ASfP или PSfP могут работать со связанным менеджером по работе с партнерами (SAM/TAM), чтобы лучше понять доступные варианты.
- Доступ к реализации базовых политик с центром партнеров можно получить через [запрос на обслуживание центра партнеров](https://partner.microsoft.com/dashboard/support/csp/servicerequests/create?category=csp). Выберите *MFA & защищенную модель приложений* в качестве раздела.

### <a name="how-do-i-get-technical-information-to-adopt-secure-application-model-framework"></a>Разделы справки получить технические сведения о том, как внедрить безопасную платформу модели приложений? 

Технические варианты поддержки для Azure Active Directory доступны по преимуществам MPN. Партнеры с доступом к активной подписке ASfP или PSfP могут работать со связанным менеджером по управлению учетными записями (SAM/TAM), чтобы лучше понять доступные варианты.
