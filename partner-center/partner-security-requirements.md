# <a name="partner-security-requirements"></a>Требования к безопасности партнерам

**Применяется к**

- Все партнеры по программе поставщика облачных решений
  - Direct счет
  - Непрямой поставщик
  - Косвенный торговый посредник
- Все поставщики панели управления
- Все помощников

Безопасность и конфиденциальность клиентов и партнеров являются главными приоритетами Майкрософт. Мы продолжаем см. в разделе возрастающий номер более сложных атак на систему безопасности, в основном связанных с скомпрометированных удостоверений. Упреждающие элементов управления играют ключевую роль в общей стратегии защиты для предотвращения атак на систему безопасности, мы начнем, принудительное применение набора требований обязательных безопасности для защиты, партнерами и клиентами.

> [!NOTE]
> Настоятельно рекомендуется всех партнеров, совершения транзакций через независимом облаке (компанией 21Vianet, Германии и государственных организаций США) действовать и немедленно принять эти новые требования безопасности. Тем не менее эти партнеры не необходимы для обеспечения соответствия новым требованиям безопасности действующие 1 июля. Корпорация Майкрософт предоставит дополнительные сведения о принудительного применения этих требований безопасности для независимых облаков в будущем.

## <a name="overview-of-the-requirements"></a>Обзор требований

Всех партнеров, участвующих в программе поставщика облачных решений, элемент управления панели поставщики и партнеры помощник по должны использовать многофакторную идентификацию (MFA) для каждого пользователя в свой клиент партнера. Это можно сделать, включив два [Azure Active Directory базовая политика](https://docs.microsoft.com/azure/active-directory/conditional-access/concept-baseline-protection). Базовая политика — это набор стандартных политик, которые позволяют защищать компании от многих распространенных атак. Этих распространенных атак может включать пароль Распылитель, воспроизведения и фишинга. Базовая политика доступны во всех выпусках Azure Active Directory. Майкрософт обеспечение эти политики защиты базовых показателей доступности для всех поскольку атак на основе удостоверений были развертывают за последние несколько лет.

В следующей таблице описаны два базовых показателей политики, которые должны быть включены.

|**Политика**| |
|-----|-----|
|**Требовать многофакторную Идентификацию для администраторов**|Для включения MFA требуется для политики "Администраторы", потребуется регистрации для многофакторной проверки Подлинности, используя приложение для проверки подлинности пользователей в роли администратора. После завершения регистрации многофакторной проверки Подлинности, администраторы должны будут выполнять эту Проверку при каждом входе.|
|**Защита конечных пользователей**|Защита конечного пользователя — рисков многофакторной проверки Подлинности базовых показателей политику, которая обеспечивает защиту всех пользователей в каталоге. При включении этой политики требует регистрации для многофакторной проверки Подлинности, используя приложение для проверки подлинности всех пользователей. Пользователи могут пропустить запрос регистрации на многофакторную Идентификацию для 14 дней, после чего они будут заблокированы вход до регистрации в MFA. После регистрации для многофакторной проверки Подлинности пользователей будут запрашиваться многофакторной проверки Подлинности только во время рискованных попытки входа в систему. Скомпрометированные учетные записи пользователей блокируются, пока не сбросить пароль и события риска были отключены.|

Если эти политики включены, каждый пользователь сможет использовать Azure MFA без дополнительной платы. Если вы используете стороннее решение, затем необходимо для принудительного применения многофакторной проверки Подлинности для каждого пользователя при доступе к Microsoft коммерческих облачных служб.

> [!IMPORTANT]
> Поскольку многофакторной проверки Подлинности будет применяться для каждого пользователя в каталог партнеров, может быть последствия для любой службы автоматизации или интеграции, который использует учетные данные пользователя. Чтобы устранить это влияние, достаточно изменить метод автоматизации или интеграции подключается к коммерческих облачных служб Майкрософт. Если вы подключаетесь к служба поддерживает токена проверки подлинности на основе, то рекомендуется реализовать [framework модель защиты приложения](https://docs.microsoft.com/partner-center/develop/enable-secure-app-model).

## <a name="what-actions-do-i-need-to-take"></a>Какие действия необходимо выполнить? 

Чтобы обеспечить защиту каждого пользователя в организации партнера по, необходимо включить [требования MFA для администраторов](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-administrators) и [защиты конечного пользователя](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-end-users) базовая политика. Перед включением эти политики, важно понимать их назначение и как они влияют на все автоматические средства или интеграции и пользователей.

### <a name="considerations"></a>Рекомендации

Поскольку требования к безопасности применяются ко всем пользователям в каталоге партнеров, некоторые рекомендации по необходимо внести, чтобы избежать проблем при развертывании. Эти рекомендации включают идентификации пользователей в Azure Active Directory, который нельзя или не следует выполнять многофакторной проверки Подлинности, а также приложений и клиентов, принятые в организации, не поддерживающих современную проверку подлинности.

#### <a name="legacy-protocols"></a>Устаревшие протоколы

Устаревшие протоколы аутентификации (IMAP, SMTP, POP3 и др.) используются почтовые клиенты для проверки подлинности запросов. Эти протоколы не поддерживают MFA. Большая часть скомпрометированных учетных записей, увиденное в Microsoft вызваны недобросовестных сторон, выполнение атаки, направленные на устаревшие протоколы, попытка обхода многофакторной проверки Подлинности. Чтобы обязательна многофакторная проверка Подлинности при входе в учетную запись в каталог партнеров, а недобросовестных сторон не могут обходить MFA, эти требования к безопасности блокирует все запросы проверки подлинности от устаревших протоколов.

### <a name="enabling-the-baseline-policies"></a>Включение политик базовых показателей

См. в разделе [реализации руководства требования к безопасности партнеров](tutorials/partner-security-requirements.yml) для интерактивное о реализации политики базовых показателей.  

#### <a name="require-mfa-for-admins"></a>Требовать многофакторную Идентификацию для администраторов

*Требования MFA для администратора* базовая политика требует многофакторной проверки Подлинности для следующих ролей каталога, считается наиболее привилегированных ролей Azure Active Directory:

- Глобальный администратор
- Администратор SharePoint
- Администратор Exchange
- Администратор условного доступа
- Администратор безопасности
- Администратор службы технической поддержки и администратор паролей
- Администратор выставления счетов
- Администратор пользователей

При включении MFA требуется для политики "Администраторы", указанных выше ролей девять администратора потребуется для регистрации для многофакторной проверки Подлинности, используя приложение для проверки подлинности. После завершения регистрации многофакторной проверки Подлинности, администраторы должны будут выполнять эту Проверку по одному их входа в систему.

Если ваша организация использует эти учетные записи используется в сценарии или код, рассмотрите возможность замены их с помощью [управляемые удостоверения](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/overview).

Чтобы включить эту политику и защитить администраторов:

1. Войдите в **портала Azure** как глобального администратора, администратора безопасности или администратора условного доступа.
2. Перейдите к **Azure Active Directory** > **условного доступа**.
3. В списке политик выберите **базовая политика: Требовать многофакторную Идентификацию для администраторов**.
4. Задайте **включить политику** для **использовать политику немедленно**.
5. Нажмите кнопку **Сохранить**.

    ![Требовать многофакторную Идентификацию для администраторов](images/security/baseline-policy-require-mfa-for-admins.png)

> [!WARNING]
> Прежде чем эта политика включена, убедитесь, что ваши пользователи не используют устаревшие протоколы аутентификации. См. в статье [как: Блок устаревшей проверки подлинности в Azure Active Directory с условным доступом](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-legacy-auth#identify-legacy-authentication-use) Дополнительные сведения.

> [!IMPORTANT]
> Имеется известная проблема, которые влияют на возможность подключения к Exchange Online PowerShell с помощью делегированные права администратора. См. в разделе [Exchange Online PowerShell](#exchange-online-powershell) известная проблема, перед включением этой политики, если вы используете этот модуль PowerShell.

#### <a name="end-user-protection"></a>Защита конечных пользователей

Базовая политика защиты конечного пользователя защищает все пользователи в каталоге. При включении этой политики всем пользователям потребуется для регистрации в Azure MFA в течение 14 дней. После регистрации пользователей будут запрашиваться многофакторной проверки Подлинности только во время рискованных попытки входа в систему. Скомпрометированные учетные записи пользователей блокируются, пока не сбросить пароль и риск увольнения.

Политика **базовая политика: Защита конечных пользователей** поставляется предварительно настроенным и будут отображаться в верхней при переходе к колонка условного доступа на портале Azure.

Чтобы включить эту политику и защитить пользователей.

1. Войдите в **портала Azure** как глобального администратора, администратора безопасности или администратора условного доступа.
2. Перейдите к **Azure Active Directory** > **условного доступа**.
3. В списке политик выберите **базовая политика: Защита конечных пользователей (Предварительная версия)** .
4. Задайте **включить политику** для **использовать политику немедленно**.
5. Нажмите кнопку **Сохранить**.

    ![Защита конечных пользователей](images/security/baseline-policy-end-user-protection.png)

> [!WARNING]
> Прежде чем эта политика включена, убедитесь, что ваши пользователи не используют устаревшие протоколы аутентификации. См. в статье [как: Блок устаревшей проверки подлинности в Azure Active Directory с условным доступом](https://docs.microsoft.com/azure/active-directory/conditional-access/howto-baseline-protect-legacy-auth#identify-legacy-authentication-use) Дополнительные сведения.

> [!IMPORTANT]
> Нет известных проблем, которые влияет на возможность подключения к Exchange Online PowerShell с помощью делегированные права администратора. См. в разделе [Exchange Online PowerShell](#exchange-online-powershell) известная проблема, перед включением этой политики, если вы используете этот модуль PowerShell.

## <a name="common-issues"></a>Общие проблемы

### <a name="azure-active-directory"></a>Azure Active Directory

#### <a name="aadsts50076"></a>AADSTS50076

После включения политики базовых показателей, может оказаться, что службу автоматизации или интеграции обнаружил исключение следующего вида

    AADSTS50076: Due to a configuration change made by your administrator, or because you moved to a new location, you must use multi-factor authentication to access 'MyApp'.

Причины исключения — что вход выполняется с использованием учетных данных пользователя и многофакторной проверки Подлинности теперь является обязательным. Чтобы устранить это исключение, необходимо использовать маркер доступа для проверки подлинности. См. в разделе [руководство по Secure модели приложения](http://assetsprod.microsoft.com/secure-application-model-guide.pdf) Дополнительные сведения.

>[!IMPORTANT]
>Большинство современных API и модули PowerShell поддерживает возможность использования маркера доступа для проверки подлинности. Тем не менее существуют некоторые, который в настоящее время не поддерживают эту функцию. Если вам нужна помощь при определении того, если модуль API или PowerShell, вы пытаетесь использовать поддерживает использование маркера доступа для проверки подлинности, отправьте сообщение на [безопасности руководство Partner Center Group](https://www.microsoftpartnercommunity.com/t5/Partner-Center-Security-Guidance/ct-p/partner-center-security-guidance) сообщества.

#### <a name="aadsts700082"></a>AADSTS700082

После реализации framework модель защиты приложения есть вероятность того, вы получите следующее исключение 90 дней после создания маркер начальной обновления

    The refresh token has expired due to inactivity. The token was issued on 2019-01-02T09:19:53.5422744Z and was inactive for 90.00:00:00

По отношению к Azure Active Directory максимальное время существования для обновления маркера составляет 90 дней. Чтобы устранить эту ошибку, необходимо будет создать и безопасно сохранить новый маркер обновления. Обратите внимание, что можно обновить маркер обновления программно, так как для каждого запроса на маркер доступа Azure Active Directory возвращается новый маркер обновления. Вы можете реализовать соответствующую логику, чтобы обновить маркер обновления, безопасно хранящимся до истечения срока его действия.

См. в разделе [Настройка времени жизни маркеров в Azure Active Directory](https://docs.microsoft.com/azure/active-directory/develop/active-directory-configurable-token-lifetimes) Дополнительные сведения.
